# Macro Bias

Macro Bias is an automated quantitative research desk. It generates a daily pre-market research report by combining a K-Nearest Neighbors historical price analog with real-time geopolitical and financial news, then distributing that synthesis as a premium sector-bias briefing.

The system is designed as a production-ready financial research SaaS rather than a personal script. It maintains a persistent market history, scores the current regime from stored cross-asset data, uses an LLM as a structured risk manager, logs every published briefing for audit and backtesting, and distributes the result through controlled outbound channels.

## Overview

Macro Bias produces a daily report with three layers:

1. A quantitative baseline derived from the latest K-NN regime score.
2. A qualitative overlay sourced from pre-market news headlines.
3. A final institutional briefing generated by an LLM under a strict JSON contract.

The output is a daily sector-bias note that tells the subscriber what the baseline math is saying, whether the current news flow confirms or threatens that baseline, and when the historical analog should no longer be trusted at face value.

## The Macro Override Engine

The Macro Override engine is the core business logic.

The K-NN model remains the baseline decision layer. It identifies historically similar sessions from persisted market history and produces the daily bias score from those analogs. That baseline is useful only when today's market structure still resembles the historical sample.

The LLM is not used as a discretionary signal generator. It acts as a risk manager. Its job is to evaluate whether the live pre-market news tape contains a structural break that invalidates the historical analog framework.

Examples of structural breaks include:

- wars, attacks, missile strikes, or ceasefires
- sanctions, tariffs, or embargoes
- central-bank surprises or abrupt policy shifts
- banking stress, credit accidents, or liquidity events
- other macro shocks that can break normal analog behavior

If the system determines that the analog is no longer safe, it sets `is_override_active = true` and marks the briefing as `MACRO OVERRIDE ACTIVE`. In practical terms, that means the user is being warned not to trade the baseline math blindly.

The legacy standalone `macro-override.ts` script has been retired. Override logic now lives inside the unified daily briefing pipeline.

## Daily System Architecture

The live orchestration boundary is the secure cron route at `/api/cron/publish`.

### Execution flow

1. Vercel Cron triggers `/api/cron/publish`.
2. The route validates the bearer secret using `CRON_SECRET` or `PUBLISH_CRON_SECRET`.
3. A same-day publish guard checks `daily_market_briefings` for the current UTC `briefing_date`.
4. If a briefing already exists for today, the route returns `200 OK` and skips generation and dispatch.
5. If no briefing exists, the route runs `upsertDailyMarketData()` to refresh prices and persist the latest quant snapshot.
6. The route loads recent rows from `macro_bias_scores`.
7. `generateDailyBriefing()` runs the quant branch and the news branch in parallel with `Promise.all(...)`.
8. Anthropic synthesizes the final report under a strict JSON schema.
9. If news or LLM generation fails, retry and fallback logic still produce a complete briefing.
10. The finished report is inserted into `daily_market_briefings`.
11. Email distribution runs through Resend, with optional secondary channels when configured.

### Pipeline diagram

```text
Vercel Cron
  -> /api/cron/publish
    -> cron authorization
    -> same-day publish guard
    -> upsertDailyMarketData()
    -> load latest macro_bias_scores context
    -> generateDailyBriefing()
       -> Promise.all(fetchNews(), getQuantScore())
       -> Anthropic synthesis or deterministic fallback
    -> persistDailyBriefing()
    -> dispatchQuantBriefing()
    -> JSON response summary
```

### Parallel execution

The daily briefing generator does not block the qualitative path on the quantitative path.

It runs these branches concurrently:

- `fetchNews()` pulls the pre-market headline set from Finnhub.
- `getQuantScore()` reconstructs the daily K-NN context from PostgreSQL-backed market history and the latest stored engine inputs.

This keeps the route fast enough for the pre-market window while preserving a clean separation between model state and news state.

### Resilience model

The generator is built to degrade gracefully rather than fail silently.

- `retry.ts` applies exponential backoff to the Finnhub and Anthropic calls.
- `daily-briefing-strategies.ts` uses a Strategy Pattern to switch between `news-aware` and `news-unavailable` briefing modes.
- If the news API fails, the system still generates a full report with a clear news-unavailable disclaimer.
- If the LLM fails, the system falls back to a deterministic institutional briefing built from the quant context and available news state.

This ensures the daily desk produces a report even when external dependencies degrade.

## Reporting Contract

The LLM is required to return strict JSON in the following shape:

```json
{
  "is_override_active": true,
  "newsletter_copy": "..."
}
```

The final briefing is designed around four fixed sections:

1. `THE BOTTOM LINE`
2. `THE REGIME PLAYBOOK`
3. `MACRO OVERRIDE STATUS`
4. `QUANT CORNER`

That stable output contract makes the reports easier to audit, test, backtest, and render consistently across delivery channels.

## Persistence And Backtesting

Every generated report is logged to `public.daily_market_briefings`.

That table stores the daily briefing date, the underlying trade date, the quant score, the bias label, the override state, the news status, headline summaries, analog reference, full briefing body, source model, generation method, and timestamp.

This ledger has two jobs:

1. Preserve an audit trail of every published research note.
2. Enable future analysis of Macro Override behavior, analog reliability, and LLM-versus-fallback output quality.

It also powers the same-day publish guard so the cron job cannot double-fire and spam the subscriber list.

## Data Model

The daily briefing path relies on three primary datasets:

- `etf_daily_prices`: raw market history used to construct the quant feature set.
- `macro_bias_scores`: persisted daily regime scores and model inputs.
- `daily_market_briefings`: the final generated report ledger used for backtesting and duplicate-run protection.

## Email Dispatch And Deliverability

Email distribution is handled by Resend.

For production delivery, the intended setup is:

- `RESEND_FROM_ADDRESS` points to an authenticated `macro-bias.com` sender
- SPF is published for the sending domain
- DKIM is enabled through Resend
- DMARC is enforced so alignment failures are observable and policy-backed

The codebase also supports safe testing behavior. During a shadow run, outbound mail is forced to `SHADOW_RUN_EMAIL` so the pipeline can be tested end-to-end without touching the live subscriber list.

If you are validating the system locally or in pre-production, keep the shadow-run recipient lock enabled. If you are moving to production, switch to the authenticated custom-domain sender and remove the shadow-run recipient override.

## Shadow Run Testing

The recommended local test path is to invoke the publish route directly with the cron secret header.

### Trigger the daily route

```bash
curl -X POST http://localhost:3000/api/cron/publish \
  -H "Authorization: Bearer <CRON_SECRET>"
```

### Expected behavior

On the first successful run of the day, the route should:

1. refresh market data
2. generate the daily briefing
3. persist the new row to `daily_market_briefings`
4. send exactly one email to `SHADOW_RUN_EMAIL`

On a second run on the same UTC date, the same-day publish guard should return `200 OK` and skip dispatch.

Example skip response:

```json
{
  "briefingDate": "2026-04-08",
  "message": "Briefing already generated for today.",
  "ok": true,
  "status": "skipped"
}
```

This is the safety mechanism that prevents accidental duplicate briefings and subscriber spam if the cron is retried or manually re-fired.

## Scheduling

The primary daily job is defined in `vercel.json`:

```json
{
  "path": "/api/cron/publish",
  "schedule": "45 12 * * *"
}
```

The schedule is expressed in UTC. Adjust operational expectations accordingly for New York pre-market timing and daylight-saving changes.

## Architectural Principles

The current codebase is organized around a few explicit principles:

- quant remains the baseline model
- the LLM is used as a structured synthesis and risk-escalation layer, not as a discretionary trading engine
- a daily report must still be generated under partial dependency failure
- every report must be persisted for auditability and backtesting
- distribution must be safe by default during testing and shadow runs

That architecture is what turns Macro Bias into a daily automated financial research SaaS rather than a one-off market script.
